Kafka Producer Pipeline
+ Kafka Consumer Pipeline

Build two separate pipelines as a real streaming architecture. Pipeline A produces synthetic click events into a Kafka topic. Pipeline B reads from that same topic, applies a 10-second tumble window with aggregation, then fans out to a Kafka output topic and a Print sink.

✓ Flink session connected ⚠ Kafka JAR required ⚠ Kafka broker running

▦ What You'll Build

Two pipelines that work together through a Kafka topic. This is the real-world pattern — a producer job writes data, and a separate consumer job processes it independently. Both jobs run simultaneously.

Concepts practised: Kafka source + sink configuration, consumer groups, tumble windows, building two separate pipelines, live preview from a Kafka reader table.

! Prerequisites

  • Kafka JAR — flink-sql-connector-kafka-x.x.x.jar must be in /opt/flink/lib/. Upload via ⊙ Systems → Upload JAR. Restart the SQL Gateway after uploading.
  • Kafka broker — a Kafka instance reachable from the Flink cluster at kafka:9092 (or update to your address). Topics raw-clicks and click-summary will be auto-created by the producer on first write.
  • Studio connected — active Flink session in the topbar.
⚠
Start Pipeline A first. The Kafka source in Pipeline B reads from a topic that Pipeline A writes. Run the producer first, wait a few seconds for data to appear, then build and submit the consumer.

Connect the nodes and set edge types

Draw these edges in order:

  • raw_clicks_source → click_window — set type: HASH
  • click_window → click_summary_sink — set type: HASH

✓ Verify & Live Preview

In Studio, click the Job Graph tab. You should see two running jobs:

[your-session] kafka-producer     → running  (Job 1)
[your-session] kafka-consumer     → running  (Job 2)

Click either job name to see its operator graph and throughput metrics.

-- TaskManager stdout shows windowed aggregation results:
docker logs flink-taskmanager --follow --tail=20

-- Rows look like:
-- +I[2026-..., 2026-..., /home, click, 42, 312.5, 890]

Create a reader table with a different consumer group to preview the output topic without interfering with the production consumer:

-- In a new Studio editor tab:
CREATE TEMPORARY TABLE click_summary_reader (
  window_start    TIMESTAMP(3),
  window_end      TIMESTAMP(3),
  page            STRING,
  action          STRING,
  click_count     BIGINT,
  avg_ms          DOUBLE,
  max_ms          INT
) WITH (
  'connector'                    = 'kafka',
  'topic'                        = 'click-summary',
  'properties.bootstrap.servers' = 'kafka:9092',
  'properties.group.id'          = 'click-preview-01',
  'scan.startup.mode'            = 'earliest-offset',
  'format'                       = 'json'
);

-- Stream results into the Results tab:
SELECT * FROM click_summary_reader;
💡
Always use a separate consumer group for previewing. The production consumer group (click-processor-01) tracks offsets for the pipeline. A different group for SELECT queries won't affect those offsets.

⟨⟩ Generated SQL

SET 'execution.runtime-mode'           = 'streaming';
SET 'parallelism.default'              = '1';
SET 'execution.checkpointing.interval' = '10000';

CREATE TABLE IF NOT EXISTS clicks_gen (
  user_id   BIGINT,
  page      STRING,
  action    STRING,
  duration  INT,
  ts        TIMESTAMP(3),
  WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH (
  'connector'       = 'datagen',
  'rows-per-second' = '20'
);

CREATE TABLE IF NOT EXISTS raw_clicks_sink WITH (
  'connector'                    = 'kafka',
  'topic'                        = 'raw-clicks',
  'properties.bootstrap.servers' = 'kafka:9092',
  'format'                       = 'json'
) LIKE clicks_gen (EXCLUDING ALL);

INSERT INTO raw_clicks_sink SELECT * FROM clicks_gen;
CREATE TABLE IF NOT EXISTS raw_clicks_source (
  user_id   BIGINT,
  page      STRING,
  action    STRING,
  duration  INT,
  ts        TIMESTAMP(3),
  WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH (
  'connector'                    = 'kafka',
  'topic'                        = 'raw-clicks',
  'properties.bootstrap.servers' = 'kafka:9092',
  'properties.group.id'          = 'click-processor-01',
  'scan.startup.mode'            = 'earliest-offset',
  'format'                       = 'json'
);

CREATE TABLE IF NOT EXISTS click_summary_sink WITH (
  'connector'                    = 'kafka',
  'topic'                        = 'click-summary',
  'properties.bootstrap.servers' = 'kafka:9092',
  'format'                       = 'json'
) LIKE raw_clicks_source (EXCLUDING ALL);

CREATE TABLE IF NOT EXISTS click_debug WITH (
  'connector' = 'print'
) LIKE raw_clicks_source (EXCLUDING ALL);

INSERT INTO click_summary_sink
SELECT
  window_start, window_end, page, action,
  COUNT(*) AS click_count,
  AVG(CAST(duration AS DOUBLE)) AS avg_ms,
  MAX(duration) AS max_ms
FROM TABLE(TUMBLE(TABLE raw_clicks_source,
  DESCRIPTOR(ts), INTERVAL '10' SECOND))
GROUP BY window_start, window_end, page, action;

INSERT INTO click_debug
SELECT
  window_start, window_end, page, action,
  COUNT(*) AS click_count
FROM TABLE(TUMBLE(TABLE raw_clicks_source,
  DESCRIPTOR(ts), INTERVAL '10' SECOND))
GROUP BY window_start, window_end, page, action;

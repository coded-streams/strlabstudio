ⓘAbout Str:::lab Studio

Str:::lab Studio is a browser-based SQL Studio that connects to any Flink SQL Gateway. It provides a complete development, monitoring, and administration experience without the overhead of a full platform deployment. Works with local Docker, Kubernetes, and any self-hosted Flink cluster.

What it is

A zero-dependency single-file HTML application served by nginx. All SQL execution is driven by the Flink SQL Gateway REST API. Results stream live into the browser. Session state is managed client-side in localStorage.

Who it's for

Streaming data engineers and platform teams who want a proper IDE experience for Apache Flink SQL — query editing, live result streaming, real-time job monitoring, checkpoint tracking, and professional reporting — all from a browser.

Open source

Created by Nestor Martourez A. A — open source under MIT. Community contributions welcome at github.com/coded-streams/strlabstudio.

Kube Operator

The Str:::lab Studio Kube Operator manages Studio deployments as first-class Kubernetes CRDs. Install via Helm from strlabstudio-operator.

⚡Quickstart

The strlabstudio image already contains nginx, the proxy config, WebDAV JAR storage, and the entrypoint. No extra containers needed — just point the env vars at your existing cluster. 

services:

  flink-studio:
    image: codedstreams/strlabstudio:latest
    container_name: StrLab-Studio-IDE
    restart: unless-stopped
    ports:
      - "3030:80"
    environment:
      # nginx proxies /flink-api/     → flink-gateway-cors-proxy:8084
      # nginx proxies /jobmanager-api → codedstream-jobmanager:8081
      FLINK_GATEWAY_HOST: flink-gateway-cors-proxy
      FLINK_GATEWAY_PORT: "8084"
      JOBMANAGER_HOST:    codedstream-jobmanager
      JOBMANAGER_PORT:    "8081"
    volumes:
      - udf-jars:/var/www/udf-jars   # UDF JAR storage (WebDAV PUT target)
    networks:
      - kafka-network-02             # same network as your Flink cluster

volumes:
  udf-jars:                          # named volume for uploaded JARs

networks:
  kafka-network-02:
    external: true                   # joins your existing network, does not recreate it
docker compose up -d
open http://localhost:3030
💡
Select "Via Studio" on the connect screen. nginx resolves flink-gateway-cors-proxy and codedstream-jobmanager by container name through kafka-network-02. No CORS issues, no browser-to-cluster connection needed.
⚠
For UDF JAR upload — you also need to mount the udf-jars volume on your existing gateway container so ADD JAR '/var/www/udf-jars/name.jar' can find the file. See the UDF Manager section for the exact restart command.

🏗Architecture

Str:::lab Studio is a static web app served by nginx. All SQL execution happens via the Flink SQL Gateway REST API. The browser manages all session state client-side. The Studio image is self-contained — nginx serves the IDE and proxies all API calls to your Flink cluster.

Request flow

Every browser request routes through the Studio nginx container. No direct browser-to-cluster connection needed.

Deployment options

How you connect Studio to your cluster depends on your setup. All options use the same Studio image — only the env vars change.

Provides a privileged view of the entire Flink cluster across all users and sessions. Access from the connect screen by selecting the Admin mode and entering the admin passcode.

Capabilities

  • Full cluster job visibility — see ALL jobs running on the cluster, not just your own session's jobs.
  • Per-session activity — Sessions Activity modal shows each session's submitted jobs, query count, and running job status.

  • Admin-grade PDF reports — Technical Report (full engineering detail: vertex metrics, JVM heap, checkpoint durations) or Business/Management Report (plain-language annotations, resource utilisation %, job health summary).
  • Change admin passcode — click the 🛡 ADMIN badge in the topbar to open Admin Settings.

🔀Full Pipeline Demo

Complete multi-stage streaming pipeline: datagen → Kafka → enrichment → windowed aggregation → live preview.

SET execution.runtime-mode = 'STREAMING';
SET parallelism.default = '4';
SET state.backend = 'rocksdb';
SET table.exec.mini-batch.enabled = 'true';
SET table.exec.mini-batch.allow-latency = '2000';
SET pipeline.name = 'trading-pipeline-demo';
CREATE TEMPORARY TABLE raw_trades (
  trade_id STRING, symbol STRING,
  amount DECIMAL(18, 2),
  ts AS CURRENT_TIMESTAMP,
  WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH (
  'connector' = 'datagen',
  'rows-per-second' = '100',
  'fields.amount.min' = '100',
  'fields.amount.max' = '999999'
);
INSERT INTO kafka_trades SELECT * FROM raw_trades;
INSERT INTO enriched_trades
SELECT trade_id, symbol, amount,
  (amount / 100000.0) AS risk_score,
  CASE WHEN (amount / 100000.0) >= 0.80 THEN 'CRITICAL'
       WHEN (amount / 100000.0) >= 0.55 THEN 'HIGH'
       WHEN (amount / 100000.0) >= 0.30 THEN 'MEDIUM'
       ELSE 'LOW' END AS risk_tier, ts
FROM kafka_trades;
INSERT INTO trade_summary
SELECT TUMBLE_START(ts, INTERVAL '10' SECOND) AS window_start,
  symbol, COUNT(*) AS trade_count,
  SUM(CASE WHEN risk_tier = 'CRITICAL' THEN 1 ELSE 0 END) AS critical_count
FROM enriched_trades
GROUP BY TUMBLE(ts, INTERVAL '10' SECOND), symbol;
SELECT trade_id, symbol, amount, risk_tier
FROM enriched_trades WHERE risk_tier IN ('CRITICAL', 'HIGH')
ORDER BY ts DESC;

ⅅUDF Manager v1.2.4 — March 2026

Complete User-Defined Function lifecycle — browsing, building, uploading, and registering — without leaving Str:::lab Studio. Access via the ⅅ UDFs button. Version 1.2.0 is a complete overhaul of the Register UDF tab with a 3-step guided wizard, confirmed fix for ClassNotFoundException, and full separate-container Docker instructions.

⚠
Two-JVM architecture — read this first JobManager (port 8081) and SQL Gateway (port 8083) are separate JVM processes. Uploading a JAR to the JobManager does not make it available in the SQL Gateway session. ADD JAR '/path/to.jar' must run inside the Gateway session with a path that exists on the Gateway container's filesystem. This was the confirmed root cause of all ClassNotFoundException errors seen in the gateway logs.

The correct tab order for Java JAR UDFs

There are 6 tabs in UDF Manager. You always work through them in this order for a JAR UDF:

1
⬡ Maven / Gradle tab — generate pom.xml or build.gradle, fill in your artifact details, build locally: mvn clean package -DskipTests → shaded JAR.
2
⇧ Upload JAR tab — do this BEFORE Register UDF — drag your shaded JAR and click Upload. Studio uploads to JobManager AND automatically runs ADD JAR in the active session. Wait for the green "✓ ADD JAR succeeded" message before proceeding.
If ADD JAR fails here — see the separate-containers fix below.
3
＋ Register UDF tab → Step 1 — click ↻ SHOW JARS to confirm your JAR appears in green. If it's empty, ADD JAR failed in step 2 — fix that first. Do not skip this check.
4
Step 2 — fill in function name, language (Java / Scala / Python), scope, and full class path. SQL preview updates live.
5
Step 3 → ⚡ Register Function — pre-flights SHOW JARS one final time, then executes CREATE TEMPORARY FUNCTION IF NOT EXISTS …. On success: "✓ registered successfully!" verified by SHOW USER FUNCTIONS.
6
📚 Library tab → ↻ Refresh — your function appears under "ⅅ User UDFs" (blue). Click it to auto-insert function_name() at the cursor.

If ADD JAR fails — separate Docker containers

⚠
Most common cause of ClassNotFoundException in Docker setups Separate jobmanager and sql-gateway services in docker-compose.yml each have their own filesystem. A JAR uploaded to the JobManager lives in /tmp/flink-web-upload/ of the JobManager container — invisible to the Gateway container.
# 1. Find your gateway container name:
docker ps

# 2. Copy JAR directly into the Gateway container:
docker cp streams-studio-udf.jar <gateway-container>:/opt/flink/usrlib/

# 3. In Register UDF → Step 1, paste the path and click ADD JAR:
ADD JAR '/opt/flink/usrlib/streams-studio-udf.jar';

Quickest fix. Does not persist across container restarts.

# In docker-compose.yml — add to BOTH services:
services:
  jobmanager:
    volumes:
      - ./udfs:/opt/flink/usrlib
  sql-gateway:
    volumes:
      - ./udfs:/opt/flink/usrlib

# Place JAR at ./udfs/ on host.
# ADD JAR path = /opt/flink/usrlib/your-udf.jar
# Persists across restarts — recommended for production.
-- If the Gateway container can reach your network:
ADD JAR 'http://your-file-server/streams-studio-udf.jar';
-- Flink 1.16+ supports HTTP URLs in ADD JAR.

Creating a UDF without a JAR — SQL Temporary View

For logic expressible in SQL (CASE WHEN, arithmetic, string functions), a Temporary View is the recommended alternative. No compilation, no upload, no ADD JAR needed.

1
▭ View Builder tab — mode: ▭ View (default).
2
Fill in the form: View Name, Source Table, Computed Columns (one per line: alias AS expression), optional WHERE.
3
Click ⚡ Create View — instant. Expected: "✓ View fraud_scored created".
4
Use it like a table — SELECT * FROM fraud_scored or as a pipeline source in INSERT INTO. Appears in Library → ▭ Session Views.

Java JAR UDF vs SQL View

⇧ Path A: Java JAR UDF

  • Use when: ML inference, regex, external API, complex stateful logic
  • Requires: Build tooling, JAR upload, ADD JAR every new session
  • Docker: Complex if containers are separate
  • Versioned: Yes — JAR is a deployable artifact
  • Persistent: No — must re-register after reconnect

▭ Path B: SQL Temporary View

  • Use when: logic fits in CASE WHEN or built-in SQL functions
  • Requires: Nothing — instant, no build step, no ADD JAR
  • Docker: Zero complexity
  • Portable: Any Flink version (1.15+)
  • Persistent: No — session-scoped (save in Project Manager)
⚠
LANGUAGE SQL does not exist in Flink FunctionLanguage.SQL is not implemented in any Flink release (1.15–1.20+). CREATE FUNCTION … LANGUAGE SQL always throws "Unrecognized function language string SQL". The language dropdown only shows Java, Scala, Python. For SQL-based reusable logic, use the ▭ View Builder tab.

CREATE TEMPORARY TABLE tower_telemetry (
    event_id           VARCHAR,
    tower_id           INT,
    tower_raw_status   INT,    -- 0=ACTIVE 1=DEGRADED 2=MAINTENANCE 3=CONGESTED 4=OFFLINE

) WITH (

    'rows-per-second'               = '50',
    'fields.tower_id.min'           = '1',
    'fields.tower_id.max'           = '200',
    'fields.tower_raw_status.min'   = '0',
    'fields.tower_raw_status.max'   = '4',
    'fields.rssi_raw.min'           = '-120.0',
    'fields.rssi_raw.max'           = '-40.0',
    'fields.sinr_raw.min'           = '0.0',
    'fields.sinr_raw.max'           = '30.0',
    'fields.latency_ms.min'         = '5',
    'fields.latency_ms.max'         = '350',
    'fields.packet_loss_pct.min'    = '0.0',

-- Tab 4: Use the Java UDF
SELECT
    event_id,
    tower_id,
    CASE tower_raw_status
        WHEN 0 THEN 'ACTIVE' WHEN 1 THEN 'DEGRADED'
        WHEN 2 THEN 'MAINTENANCE' WHEN 3 THEN 'CONGESTED'
        ELSE 'OFFLINE'
    END                                          AS tower_status,
    ROUND(rssi_raw, 1)                           AS rssi_dbm,
    ROUND(packet_loss_pct, 2)                    AS packet_loss_pct,
    enrich_severity(packet_loss_pct / 15.0)      AS alert_severity,
    active_subscribers,
    event_ts
FROM tower_telemetry LIMIT 20;

-- Expected: CRITICAL when packet_loss_pct > 12.0 (>= 80% of 15)

▭
No JAR required for this path. You can also build this view visually: UDF Manager → ▭ View Builder → fill in the form → ⚡ Create View.
-- Tab 5: Create SQL View (no JAR, no ADD JAR, no compilation)
CREATE TEMPORARY VIEW signals_enriched AS
SELECT
    event_id,
    tower_id,
    CASE tower_raw_status
        WHEN 0 THEN 'ACTIVE' WHEN 1 THEN 'DEGRADED'
        WHEN 2 THEN 'MAINTENANCE' WHEN 3 THEN 'CONGESTED'
        ELSE 'OFFLINE'
    END                                                           AS tower_status,
    ROUND(rssi_raw, 1)                                            AS rssi_dbm,
    ROUND(sinr_raw, 1)                                            AS sinr_db,
    latency_ms,
    ROUND(packet_loss_pct, 2)                                     AS packet_loss_pct,
    active_subscribers,
    CASE
        WHEN rssi_raw > -70 AND sinr_raw > 20 AND packet_loss_pct < 1 THEN 'EXCELLENT'
        WHEN rssi_raw > -85 AND sinr_raw > 10 AND packet_loss_pct < 3 THEN 'GOOD'
        WHEN rssi_raw > -100 AND sinr_raw > 5  AND packet_loss_pct < 8 THEN 'FAIR'
        ELSE 'POOR'
    END                                                           AS signal_quality,
    CASE
        WHEN packet_loss_pct / 15.0 >= 0.80 THEN 'CRITICAL'
        WHEN packet_loss_pct / 15.0 >= 0.55 THEN 'HIGH'
        WHEN packet_loss_pct / 15.0 >= 0.30 THEN 'MEDIUM'
        ELSE 'LOW'
    END                                                           AS alert_severity,
    CASE
        WHEN latency_ms < 20  AND packet_loss_pct < 0.5 THEN 'GOLD'
        WHEN latency_ms < 50  AND packet_loss_pct < 1.0 THEN 'SILVER'
        WHEN latency_ms < 100 AND packet_loss_pct < 3.0 THEN 'BRONZE'
        ELSE 'BREACHED'
    END                                                           AS sla_class,
    event_ts
FROM tower_telemetry;

SHOW VIEWS; -- should list signals_enriched

v1.2.4March 2026current
  • nginx WebDAV PUT HTTP 500 fixed — JAR upload was failing with 500 because /var/www/udf-jars/ was created by root but written to by the nginx worker process (user nginx). Fixed with chown nginx:nginx in docker-entrypoint.sh. Also switched from alias to root in the /udf-jars/ location — alias has a known incompatibility with dav_methods on nginx:alpine.
  • Via Studio connection (HTTP 500) documented — "Via Studio" mode routes API calls through the nginx proxy. This requires the Studio container to be on the same Docker network as the Flink cluster. Running the Studio container standalone (without --network) will produce HTTP 500. Use Direct Gateway mode when running standalone, or add the Studio container to the Flink Docker Compose network.
  • nginx startup "host not found" fixed — nginx resolves proxy_pass hostnames at startup by default. Variable upstreams (set $upstream) defer resolution to request time. Combined with reading the DNS resolver from /etc/resolv.conf at container startup via awk — works in Docker (127.0.0.11), Kubernetes (cluster DNS), and local environments.
  • RESOLVER export bug fixed — RESOLVER shell variable was not exported before being passed to envsubst, causing ${RESOLVER} to be substituted as empty string → resolver ; → nginx no name servers defined crash. Fixed with export RESOLVER.

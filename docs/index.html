ⓘAbout Str:::lab Studio

Str:::lab Studio is a browser-based SQL Studio that connects to any Flink SQL Gateway. It provides a complete development, monitoring, and administration experience without the overhead of a full platform deployment. Works with local Docker, Kubernetes, and any self-hosted Flink cluster.

What it is

A zero-dependency single-file HTML application served by nginx. All SQL execution is driven by the Flink SQL Gateway REST API. Results stream live into the browser. Session state is managed client-side in localStorage.

Who it's for

Streaming data engineers and platform teams who want a proper IDE experience for Apache Flink SQL — query editing, live result streaming, real-time job monitoring, checkpoint tracking, and professional reporting — all from a browser.

Open source

Created by Nestor Martourez A. A — open source under MIT. Community contributions welcome at github.com/coded-streams/strlabstudio.

Kube Operator

The Str:::lab Studio Kube Operator manages Studio deployments as first-class Kubernetes CRDs. Install via Helm from strlabstudio-operator.

⚡Quickstart

The strlabstudio image already contains nginx, the proxy config, WebDAV JAR storage, and the entrypoint. No extra containers needed — just point the env vars at your existing cluster. 

services:

  flink-studio:
    image: codedstreams/strlabstudio:latest
    container_name: StrLab-Studio-IDE
    restart: unless-stopped
    ports:
      - "3030:80"
    environment:
      # nginx proxies /flink-api/     → flink-gateway-cors-proxy:8084
      # nginx proxies /jobmanager-api → codedstream-jobmanager:8081
      FLINK_GATEWAY_HOST: flink-gateway-cors-proxy
      FLINK_GATEWAY_PORT: "8084"
      JOBMANAGER_HOST:    codedstream-jobmanager
      JOBMANAGER_PORT:    "8081"
    volumes:
      - udf-jars:/var/www/udf-jars   # UDF JAR storage (WebDAV PUT target)
    networks:
      - kafka-network-02             # same network as your Flink cluster

volumes:
  udf-jars:                          # named volume for uploaded JARs

networks:
  kafka-network-02:
    external: true                   # joins your existing network, does not recreate it
docker compose up -d
open http://localhost:3030

🏗Architecture

Str:::lab Studio is a static web app served by nginx. All SQL execution happens via the Flink SQL Gateway REST API. The browser manages all session state client-side.

Request flow

Every browser request routes through the Studio nginx container. No direct browser-to-cluster connection needed.

Deployment options

How you connect Studio to your cluster depends on your setup. All options use the same Studio image — only the env vars change.

🛡Admin Session

Provides a privileged view of the entire Flink cluster across all users and sessions. Access from the connect screen by selecting Admin mode and entering the admin passcode.

Capabilities

  • Full cluster job visibility — see ALL jobs running on the cluster, not just your own session's jobs.

  • Change admin passcode — click the 🛡 ADMIN badge in the topbar to open Admin Settings.

🔀Full Pipeline Demo

SET execution.runtime-mode = 'STREAMING';
SET parallelism.default = '4';
SET state.backend = 'rocksdb';
SET table.exec.mini-batch.enabled = 'true';
SET table.exec.mini-batch.allow-latency = '2000';
SET pipeline.name = 'trading-pipeline-demo';
CREATE TEMPORARY TABLE raw_trades (
  trade_id STRING, symbol STRING,
  amount DECIMAL(18, 2),
  ts AS CURRENT_TIMESTAMP,
  WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH (
  'connector' = 'datagen',
  'rows-per-second' = '100',
  'fields.amount.min' = '100',
  'fields.amount.max' = '999999'
);
INSERT INTO kafka_trades SELECT * FROM raw_trades;
INSERT INTO enriched_trades
SELECT trade_id, symbol, amount,
  (amount / 100000.0) AS risk_score,
  CASE WHEN (amount / 100000.0) >= 0.80 THEN 'CRITICAL'
       WHEN (amount / 100000.0) >= 0.55 THEN 'HIGH'
       WHEN (amount / 100000.0) >= 0.30 THEN 'MEDIUM'
       ELSE 'LOW' END AS risk_tier, ts
FROM kafka_trades;
INSERT INTO trade_summary
SELECT TUMBLE_START(ts, INTERVAL '10' SECOND) AS window_start,
  symbol, COUNT(*) AS trade_count,
  SUM(CASE WHEN risk_tier = 'CRITICAL' THEN 1 ELSE 0 END) AS critical_count
FROM enriched_trades
GROUP BY TUMBLE(ts, INTERVAL '10' SECOND), symbol;
SELECT trade_id, symbol, amount, risk_tier
FROM enriched_trades WHERE risk_tier IN ('CRITICAL', 'HIGH')
ORDER BY ts DESC;

ⅅUDF Manager v1.3.0 — March 2026

Complete User-Defined Function lifecycle — browsing, building, uploading, and registering — without leaving Str:::lab Studio. Access via the ⅅ UDFs button.

⚠
Two-JVM architecture — read this first JobManager (port 8081) and SQL Gateway (port 8083) are separate JVM processes. ADD JAR '/path/to.jar' must run inside the Gateway session with a path that exists on the Gateway container's filesystem. A browser URL like http://localhost:3030/udf-jars/name.jar is not a filesystem path and will always fail.
📌
ADD JAR path — container filesystem only The path entered in Step 1 must exist on the SQL Gateway container's filesystem, not in a browser. The correct path depends on your deployment:
  • Studio Docker (shared volume): /var/www/udf-jars/your-udf.jar — this is set automatically after upload
  • Kubernetes / docker cp: /opt/flink/usrlib/your-udf.jar
  • Flink lib dir (all jobs): /opt/flink/lib/your-udf.jar
  • JobManager web upload: /tmp/flink-web-upload/your-udf.jar

If you see "The path 'http://...' does not exist on the SQL Gateway container filesystem" — you entered a browser URL. Use the path chips in Step 1 to set the correct prefix.

The correct tab order for Java JAR UDFs

1
⬡ Maven / Gradle tab — generate pom.xml or build.gradle, build locally: mvn clean package -DskipTests → shaded JAR.
2
⇧ Upload JAR tab — drag your shaded JAR and click Upload. Studio uploads to the shared volume AND automatically runs ADD JAR in the active session. Wait for the green ✓ message before proceeding. The container path is set automatically.
3
＋ Register UDF tab → Step 1 — click ↻ SHOW JARS to confirm your JAR appears in green. The path input will already be pre-filled with the correct container path from step 2. If SHOW JARS is empty, ADD JAR failed — fix that first.
4
Step 2 — fill in function name, language (Java / Scala / Python), scope, class path. SQL preview updates live.
5
Step 3 → ⚡ Register Function — validates language against actual classpath (SHOW JARS), then executes CREATE TEMPORARY FUNCTION IF NOT EXISTS …. Selecting Python or Scala when a Java JAR is loaded is blocked with a clear error.
6
📚 Library tab → ↻ Refresh — your function appears under "ⅅ User UDFs" (blue). Click to auto-insert function_name() at cursor.

Gateway Volume Setup

⚠
Most common cause of ADD JAR failures in Docker setups Separate jobmanager and sql-gateway services each have their own filesystem. A JAR uploaded to the JobManager lives in /tmp/flink-web-upload/ of the JobManager container — invisible to the Gateway container.
# Copy JAR directly into the Gateway container:
docker cp your-udf.jar <gateway-container>:/opt/flink/usrlib/

# In Register UDF → Step 1, use the /opt/flink/usrlib/ chip and ADD JAR:
ADD JAR '/opt/flink/usrlib/your-udf.jar';
# In docker-compose.yml — add to BOTH services:
services:
  jobmanager:
    volumes:
      - ./udfs:/opt/flink/usrlib
  sql-gateway:
    volumes:
      - ./udfs:/opt/flink/usrlib

# ADD JAR path = /opt/flink/usrlib/your-udf.jar
-- Flink 1.16+ supports HTTP URLs in ADD JAR:
ADD JAR 'http://your-file-server/your-udf.jar';

Language validation

Step 3 runs SHOW JARS first to get the real current classpath, then validates language before executing. Selecting Python or Scala when a Java-compiled JAR is on the classpath is blocked — Flink will appear to register the function but throw ClassNotFoundException on the first SELECT.

⚠
LANGUAGE SQL does not exist in Flink FunctionLanguage.SQL is not implemented in any Flink release (1.15–1.20+). Use the ▭ View Builder tab for SQL-based reusable logic.

DESCRIBE on functions

Flink SQL Gateway throws "Table or view not found" when DESCRIBE fn_name is run on a UDF — it treats it as a table lookup. The UDF Manager automatically rewrites DESCRIBE fn_name and DESC fn_name to SHOW CREATE FUNCTION fn_name. Use SHOW USER FUNCTIONS to list all registered UDFs.

Session management

All UDF state is session-scoped. When a session expires or you disconnect, the uploaded JAR name and path are cleared from the UI automatically — you will not see a stale JAR name that no longer exists in the new session. Re-run ADD JAR and CREATE FUNCTION at the start of each new session.

Java JAR UDF vs SQL View

⇧ Path A: Java JAR UDF

  • Use when: ML inference, regex, external API, complex stateful logic
  • Requires: Build tooling, JAR upload, ADD JAR every new session
  • Docker: Requires shared volume between Gateway and Studio
  • K8s: Mount PVC or use init container to place JAR

▭ Path B: SQL Temporary View

  • Use when: logic fits in CASE WHEN or built-in SQL functions
  • Requires: Nothing — instant, no build step, no ADD JAR
  • Docker / K8s / cloud: Zero complexity
  • Persistent: No — session-scoped (save in Project Manager)

＋ Register UDF tab — 3-step guided wizard

  • Step 1 (Classpath) — SHOW JARS check. ADD JAR input with deployment-labelled path chips: /var/www/udf-jars/ (Studio Docker), /opt/flink/usrlib/ (Kubernetes / docker cp). Input rejects browser URLs and auto-corrects to the container path. This step must pass before Step 3 will succeed.

  • Step 3 (Execute) — runs SHOW JARS to get real classpath, validates language (Python/Scala blocked when Java JAR loaded), then runs CREATE FUNCTION. Human-readable error messages for all known failure modes.

USE CATALOG default_catalog;
USE `default`;
SET 'execution.runtime-mode'              = 'streaming';
SET 'parallelism.default'                 = '2';
SET 'table.exec.mini-batch.enabled'       = 'true';
SET 'table.exec.mini-batch.allow-latency' = '500 ms';
SET 'table.exec.mini-batch.size'          = '500';
SET 'table.exec.state.ttl'                = '3600000';
SET 'table.exec.source.idle-timeout'      = '15000';
CREATE TEMPORARY TABLE tower_telemetry (
    event_id           VARCHAR,
    tower_id           INT,
    tower_raw_status   INT,
    rssi_raw           DOUBLE,
    sinr_raw           DOUBLE,
    latency_ms         INT,
    packet_loss_pct    DOUBLE,
    active_subscribers INT,
    event_ts           TIMESTAMP(3),
    WATERMARK FOR event_ts AS event_ts - INTERVAL '5' SECOND
) WITH (
    'connector'                     = 'datagen',
    'rows-per-second'               = '50',
    'fields.tower_id.min'           = '1',
    'fields.tower_id.max'           = '200',
    'fields.tower_raw_status.min'   = '0',
    'fields.tower_raw_status.max'   = '4',
    'fields.rssi_raw.min'           = '-120.0',
    'fields.rssi_raw.max'           = '-40.0',
    'fields.sinr_raw.min'           = '0.0',
    'fields.sinr_raw.max'           = '30.0',
    'fields.latency_ms.min'         = '5',
    'fields.latency_ms.max'         = '350',
    'fields.packet_loss_pct.min'    = '0.0',
    'fields.packet_loss_pct.max'    = '15.0',
    'fields.active_subscribers.min' = '0',
    'fields.active_subscribers.max' = '2000'
);
SELECT tower_id, rssi_raw, sinr_raw, latency_ms, packet_loss_pct
FROM tower_telemetry LIMIT 5;

CREATE TEMPORARY FUNCTION IF NOT EXISTS enrich_severity
AS 'com.streamsstudio.udf.AlertSeverityEnricher'
LANGUAGE JAVA;

-- Confirm
SHOW USER FUNCTIONS;
SELECT
    event_id, tower_id,
    CASE tower_raw_status
        WHEN 0 THEN 'ACTIVE' WHEN 1 THEN 'DEGRADED'
        WHEN 2 THEN 'MAINTENANCE' WHEN 3 THEN 'CONGESTED'
        ELSE 'OFFLINE'
    END                                          AS tower_status,
    ROUND(packet_loss_pct, 2)                    AS packet_loss_pct,
    enrich_severity(packet_loss_pct / 15.0)      AS alert_severity,
    event_ts
FROM tower_telemetry LIMIT 20;
CREATE TEMPORARY VIEW signals_enriched AS
SELECT
    event_id, tower_id,
    CASE tower_raw_status
        WHEN 0 THEN 'ACTIVE' WHEN 1 THEN 'DEGRADED'
        WHEN 2 THEN 'MAINTENANCE' WHEN 3 THEN 'CONGESTED'
        ELSE 'OFFLINE'
    END                                                           AS tower_status,
    ROUND(rssi_raw, 1)  AS rssi_dbm,
    ROUND(sinr_raw, 1)  AS sinr_db,
    latency_ms,
    ROUND(packet_loss_pct, 2)                                     AS packet_loss_pct,
    active_subscribers,
    CASE
        WHEN rssi_raw > -70 AND sinr_raw > 20 AND packet_loss_pct < 1 THEN 'EXCELLENT'
        WHEN rssi_raw > -85 AND sinr_raw > 10 AND packet_loss_pct < 3 THEN 'GOOD'
        WHEN rssi_raw > -100 AND sinr_raw > 5  AND packet_loss_pct < 8 THEN 'FAIR'
        ELSE 'POOR'
    END                                                           AS signal_quality,
    CASE
        WHEN packet_loss_pct / 15.0 >= 0.80 THEN 'CRITICAL'
        WHEN packet_loss_pct / 15.0 >= 0.55 THEN 'HIGH'
        WHEN packet_loss_pct / 15.0 >= 0.30 THEN 'MEDIUM'
        ELSE 'LOW'
    END                                                           AS alert_severity,
    CASE
        WHEN latency_ms < 20  AND packet_loss_pct < 0.5 THEN 'GOLD'
        WHEN latency_ms < 50  AND packet_loss_pct < 1.0 THEN 'SILVER'
        WHEN latency_ms < 100 AND packet_loss_pct < 3.0 THEN 'BRONZE'
        ELSE 'BREACHED'
    END                                                           AS sla_class,
    event_ts
FROM tower_telemetry;
SHOW VIEWS;
SELECT tower_id, tower_status, rssi_dbm, latency_ms,
       packet_loss_pct, signal_quality, alert_severity, sla_class, event_ts
FROM signals_enriched LIMIT 30;

🎨Colour Describe Updated — v1.3.0

User-defined live row highlighting applied to streaming result rows in real time. Click 🎨 Colour Describe in the Results toolbar to open the configuration modal.

🎨
Fully user-controlled — no auto-detect. The field dropdown is populated from the actual rendered <th> headers of the result table. There are no hardcoded column names. This means it works identically for any pipeline, any query, any column naming convention.

Modal workflow

1

Select a query slot

The dropdown lists only SELECT result slots. SHOW, CREATE, SET, and DESCRIBE results are excluded — only plottable data queries appear.

2

Pick a field

The Field dropdown is populated from the actual rendered column headers of that query's result table — no configuration needed, works for any schema.

3

Set operator and value

Operators: == != > >= < <= contains starts ends regex. Numeric comparison is automatic when value is a number.

4

Pick a colour and style mode

8 preset swatches + custom hex picker. Style modes: Row background, Left border accent, or Text colour.

5

Add, reorder, delete rules

Rules are evaluated top-to-bottom — first match wins per row. Use ↑↓ arrows to reorder priority.

6

Apply & Activate

Click ⚡ Apply & Activate. All existing rows are immediately re-coloured. New rows are coloured as they stream in. A colour legend appears below the table with a Turn off button.

PDF report integration

When Colour Describe is active during a session, the Session Report PDF includes a Colour Describe legend section listing all active rules with field, operator, value, colour, style, and label. This lets you share a colour-annotated view of your streaming results in a static document.

⚡Flink 2.0 Compatibility v1.3.0

Str:::lab Studio connects to the Flink SQL Gateway REST API and executes all SQL server-side. Because it does not compile against Flink internals, it is compatible with Flink 1.16 through 2.x — including the officially released Flink 2.0.0 (March 2025), 2.0.1, and the upcoming 2.1/2.2 series.

✅
Studio works with Flink 2.0+ today. The SQL Gateway REST API is preserved and enhanced in Flink 2.0. Session management, ADD JAR, CREATE FUNCTION, and all streaming SQL features work identically.

🦘Otter Streams — Real-time ML Inference Integration

Otter Streams is an open-source Java SDK — built by the same author — for running real-time ML model inference inside Apache Flink streaming pipelines. It supports ONNX, TensorFlow, PyTorch, XGBoost, and PMML, and ships an otter-stream-sql module that registers ML inference as a Flink UDF. Once the JAR is uploaded via Studio's UDF Manager, you call inference directly from any SELECT query.

What it does

Runs ML models inside Flink TaskManagers as UDFs — no external inference service required. Built-in async execution, batching, result caching, GPU acceleration, Flink metrics integration, and support for cloud ML endpoints (SageMaker, Vertex AI, Azure ML).

Why it integrates with Studio

The otter-stream-sql module registers inference as a standard Flink ScalarFunction. Upload via UDF Manager, register with LANGUAGE JAVA, then call it from any SQL tab — live inference results stream into the Results panel and can be highlighted with Colour Describe rules.

Supported model frameworks

🔶

otter-stream-onnx

High-performance ONNX Runtime with GPU acceleration. Accepts PyTorch, TensorFlow, and scikit-learn exports converted to ONNX.

🔶

otter-stream-tensorflow

Native TensorFlow SavedModel integration with automatic input/output signature discovery.

🔶

otter-stream-pytorch

PyTorch TorchScript inference via Deep Java Library with automatic GPU detection.

🔶

otter-streams-xgboost

Gradient boosting inference via XGBoost4J — ideal for fraud scoring, anomaly detection, and risk classification on tabular data.

🔶

otter-stream-remote

HTTP/gRPC clients for cloud ML endpoints: AWS SageMaker, GCP Vertex AI, Azure ML.

🔶

otter-stream-pmml

PMML model support via JPMML for portable deployment across platforms.

Integration walkthrough — Studio + Otter Streams

The full workflow uses Studio's existing UDF Manager. No new tooling needed.

1
Build the shaded JAR — Add ml-inference-core and your chosen framework module to your pom.xml. Flink deps use provided scope. Build with mvn clean package -DskipTests. Include the model file in the JAR resources or place it on a path accessible to all TaskManagers.
2
Upload via Studio → Upload JAR tab — Drag the shaded JAR into UDF Manager. Studio uploads it to the shared volume and runs ADD JAR automatically. Wait for the green ✓ ADD JAR succeeded message.
3
Register UDF tab → Step 1 — Click SHOW JARS to confirm the JAR is on classpath. Step 2: class = com.codedstreams.otter.MLInferenceUDF, language = Java, scope = Temporary. Name it something descriptive (e.g. fraud_score). Step 3: Click ⚡ Register Function.
4
Call inference in SQL — The function is now usable in any SELECT tab. Write it like a built-in function against a live streaming source table.
5
Colour Describe rules — Click 🎨 Colour Describe. Select the inference output column (e.g. fraud_score), add rules (>= 0.8 → Red, >= 0.5 → Amber), activate. Inference scores are highlighted live as rows stream in.

        ELSE 'MEDIUM'

FROM transactions

📝Changelog

v1.3.0March 2026current
  • ADD JAR browser URL bug fixed — The "Use →" button in the Upload JAR list was inserting the browser URL (http://localhost:3030/udf-jars/name.jar) into Step 1 instead of the container filesystem path. Flink's SQL Gateway resolves ADD JAR paths on the Gateway container's filesystem — not browser-accessible URLs. Fixed: the button now sets the correct container path (/var/www/udf-jars/name.jar for Studio Docker). If a browser URL is entered manually into the Step 1 input, the field auto-corrects to the container path and explains why.
  • Deployment-aware container paths — All hardcoded /var/www/udf-jars/ references replaced with _getContainerJarPath(). Step 1 path chips now labelled by deployment: /var/www/udf-jars/ (Studio Docker shared volume), /opt/flink/usrlib/ (Kubernetes PVC / docker cp), /opt/flink/lib/ (Flink lib dir), /tmp/flink-web-upload/ (JobManager upload). A warning line explains the browser-URL vs container-path distinction for all deployment modes.
  • Language mismatch validates after SHOW JARS — Previously the Python/Scala language check ran before SHOW JARS, using a potentially stale _sessionJarPaths cache. It now runs after SHOW JARS fetches the real current classpath. Selecting Python or Scala when a Java JAR is on the classpath is a hard block — the registration is rejected before any SQL runs.
  • Duplicate JAR upload prevention — Before uploading, Studio checks the existing JAR list. If the same filename exists, the user is prompted to confirm overwrite. Cancelling pre-fills the existing container path in Step 1 so the user can click ADD JAR directly.
  • JAR state clears on session expiry and disconnect — _lastUploadedJarPath, _lastUploadedJarName, and _sessionJarPaths are cleared when the session expires or the user disconnects. Previously a stale JAR name from the previous session remained visible in the Upload JAR tab — clicking "Use →" on it would fail with "not found" in the new session.
  • DESCRIBE/DESC on functions auto-rewritten — Flink SQL Gateway throws "Table or view with identifier '...' doesn't exist" when DESCRIBE fn_name targets a UDF. The UDF Manager _runQ now detects DESCRIBE fn_name (no dot separator = not a catalog.db.table path) and rewrites it to SHOW CREATE FUNCTION fn_name automatically.
  • Colour Describe redesigned as user-rules-only modal — Semantic auto-colour (SEMANTIC_RULES, toggleSmartColoring) completely removed. All highlighting is user-defined. Field dropdown populated from actual rendered <th> DOM headers — no hardcoded column names, works for any pipeline, any schema.
  • Chart Report and Colour Describe slot selectors filter to SELECT-only queries — SHOW TABLES, CREATE TABLE, SET, DESCRIBE results are excluded from both slot selectors. Only plottable data queries (SELECT) appear.
